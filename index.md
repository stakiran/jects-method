# jects method
[English Version](index_EN.md) | [GitHub](https://github.com/stakiran/jects-method)

## 概要
自分の、自分による、自分のための情報管理は案外難しいものです。GTD はボリューミーで難解ですし、PARA メソッドは階層的で好き嫌いが分かれます。その他の、PKM と呼ばれるジャンルも同様です。自力でつくることもできなくはないし、楽しいものですが、素人が太刀打ちできるものではありません。自分なりのシステムにたどり着くまで通常 10 年かかります。

これではあんまりなので、この私がシンプルな概念をつくりました。比較的誰でもスタートラインに立てます。

その名も jects method。これは情報管理手法です。情報管理を 6 つの -jecst で捉えるモデルと、これをどう回すかという運用から成ります。

## モデル

### 全体像
次のようになっています。入力の口が1つ、出力の口が2つ、箱が1つありますね。

![](images/260107_162325.png)

つまり、jects method では「箱に一つの入口と二つの出口があること」と「箱の中にも小箱が三つあること」を考えます。この **合計 6 つの概念を、あなたなりに実装すればいい** のです。この 6 つだけで情報管理は網羅しきれます。

次に詳細を見ていきましょう。

### inject
入口です。箱に入れる部分を示しています。

![](images/260107_163538.png)

inject は object につながっています。つまりあなたは **何をどのように object に入れるか** を考えねばなりません。別の言い方をすると、それだけ考えればいい。入れるものと入れないものをあなたなりに決めればいいのです。

少なくとも「何でも入れる」は成立しません。箱の運用はあなたのキャパシティにかかっていますから、あなたができる程度に留めねばならないのです。この点を理解せず、自分のキャパシティと向き合わず、何でもかんでも入れようとする人があまりに多い。逃げてはいけない。自分と向き合うのです。

### object, subject, project
箱です。小箱が 3 つあります。

![](images/260107_164047.png)

object は未処理の事柄を入れる小箱です。inject から送られてくる入口でもあります。GTD でいうインボックスであり、メールで言えば受信箱です。「未処理のもの」を示します。

とにかく、まずは object に全部ぶちこみます。この考え方も GTD や受信箱と同じですが、そういうものです。この **「まずは未処理として溜める」の考え方から逃げてはいけない** のです。

subject は、方向付けに関する事柄を入れる小箱です。以下例:

- ビジョン
- モットー
- 心がけたいこと
- 昨日の反省内容
- PARA メソッドでいう AoR（Area of Responsibility）
    - たとえば「健康になりたい」なら、AoR は「健康になりたい」です。この AoR を日々眺めることで、健康になるために必要なタスクや行動を日々つくると考えます。私はタスクを生み出す源ということで「タスクソース」と呼んでいます

**subject は眺めるもの** です。眺めて日々タスクに落とすなり、行動に移すなり、頭に強く留めておいて適切なタイミングで発揮できるようにしたりなどします。行動を伴うがゆえに、ただの暗記と試験のようにはいきません。どれだけの subject を抱えて、どのように読み返すか、また練習するかは完全に人それぞれであり、あなたなりのやり方を確立せねばなりません。

project は実施が確定した事柄を入れる小箱です。以下例:

- タスク
- アクション
- TODO
- プロジェクト

表現は無数にあるので気にしなくていいです。また他人に言われたとか、自分がやりたいとか、重要性やビジネスインパクトはどうだとかいったことは関係がなく、**あなたが「確定した」と捉えたものはすべて project に入ります**。逆を言えば、それだけです。あなたが確定しなかったものは、project としては扱ってはなりません。つまりあなた自身の意思決定が大事なのです。この本質に気付き、実践しない限り、情報管理は行えません。そして、そのための判断軸は subject としてメンテナンスします。

もう一つ、project をどう扱えばいいかについても、あなた自身が考えねばなりません。このジャンルはタスク管理と呼ばれます。すでに私が整理しているのでよろしければどうぞ（[日本語](https://zenn.dev/sta/books/taskmanagement-kamikudaku) [英語](https://zenn.dev/sta/books/tasukami-english)）。

### eject
出口です。特に他者に伝えるための出口です。

![](images/260107_170149.png)

カジュアルなコミュニケーションからフォーマルな発表、もちろん発信やアップロードまで、あらゆる提出を含みます。**どんな提出方法があって、いつ何を使うかをあなた自身が考えなくてはなりません**。

また **eject は project とは分けて考えるべき** です。eject は人に影響を与えるということであり、非常に特殊なタスクなのです。だからこそ分けています。project は、そうではない大多数の事柄を広く扱います。

### reject
出口です。eject とは違って他者には見せません。単に捨てます。自分のシステムの中で生きることを拒否して追放するニュアンスです。

![](images/260107_170138.png)

これは自己啓発でよく言われる **何をやらないか** とか **何を捨てるか** といった意思決定にも等しいです。もちろん決めただけではダメで、**実際に捨てる行動が必要** です。

ただし reject も入れ物の一つであり、いわばゴミ箱に相当します。つまり **捨てたものを溜めます**。ゴミ箱と同様、あとで戻すこともできます。

## 運用
- 1: 情報のうち、jects method で管理したいものを inject に入れます
- 2: inject に入れたものは、最終的に object に移すようにします
- ===
- 3: 3 つの箱を処理します
    - object に入っているものは随時、または定期的に処理します
    - project に入っているものは日々処理していくはずです
    - subject に入っているものはこまめに処理します
- 4: 新たに生じた事柄を処理します
    - 捨てていいものは reject に入れます
    - 外に出せそうなものは eject に入れます
    - 箱に入りそうなら、3 つの小箱のいずれかに入れます
    - どこにも入らないものは捨てます（reject にも入れません）
- ===
- 5: reject を処理します
    - reject の中身を見て、問題がなければ完全に捨てます（ゴミ箱を空にする操作に等しい）
- 6: eject を処理します
    - eject の中身を見て、外に出せそうなものについて具体的に取り組みを始めたり管理したりします

## サマリー
- 自分の、自分による、自分のための情報管理をつくるための手法
- 6 つの ject で捉えて、各々自分なりに設計すればいい
- 以下の流れと特別を意識すべし
    - inject の形で箱に入れる対象を選定
    - inject からは必ず object につなぐ
    - object, subject, project の 3 つの箱の連携。いずれも内部的
    - reject の形でゴミ箱機能を実現
    - eject の形で外部への影響を管理。外部的であり、project とも分けて考えるべき

大前提として、自分なりの情報管理をつくること自体は非常にハードである。しかし jects method さえ押さえておけば迷うことはない。道のりは非常に険しいかもしれないし、おそらく可能性が高いだろうが。
